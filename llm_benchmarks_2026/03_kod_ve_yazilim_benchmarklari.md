# 3. Kod ve Yazılım Geliştirme Benchmarkları — Kapsamlı Rehber

**Tarih:** 21 Temmuz 2026

Bu belge, [Güncel LLM Benchmarkları: 2026 İçin Kapsamlı Rehber](../llm_benchmarks_guide_2026.md) ana dokümanının kod bölümünü genişleten, bağımsız okunabilir bir alt dosyadır. Aynı etiketleme kuralı geçerlidir:

- **Beklenen cevap**, veri kümesindeki doğru cevap/testtir, yani *gold answer*'dır.
- **Model çıktısı**, puanlama mekanizmasını göstermek için verilmiş somut (gerçek veya gerçekçi) bir model cevabıdır.
- **"Yayımlanmış sonuç"** denildiğinde, bir makalede, teknik raporda, resmî duyuruda veya güncel bir liderlik tablosunda gerçekten raporlanmış bir sayıdan söz edilir; kaynağı belirtilir.
- Bazı benchmarkların soruları paylaşılmadığından bazı bölümlerde gerçek görev biçimine sadık **temsilî** örnekler kullanılmıştır ve açıkça işaretlenmiştir.
- Üçüncü taraf liderlik tabloları (aider.chat, llm-stats.com, benchlm.ai vb.) hızlı değişir ve bazen birbirinden farklı sayılar verir; bu belgede bu tür sayılar tarihiyle birlikte ve "liderlik tablosu görüntüsü" olarak nitelenerek aktarılmıştır — akademik makalede raporlanmış sabit bir sonuçla karıştırılmamalıdır.

# 1. HumanEval

HumanEval (OpenAI, 2021), modele bir Python fonksiyon imzası ve docstring açıklaması verir. Model fonksiyonun gövdesini tamamlar; kod, veri kümesiyle birlikte gelen **gizli birim testlerinde** çalıştırılır. Orijinal veri kümesi **164** el yazımı programlama problemi içerir.

## 1.1. Gerçek görev türü

Yayımlanan HumanEval örneklerinden biri şu işleve benzer:

```python
def has_close_elements(numbers, threshold):
    """
    Listedeki herhangi iki sayı arasındaki fark
    threshold değerinden küçükse True döndür.
    """
```

Beklenen davranış:

```python
has_close_elements([1.0, 2.0, 3.0], 0.5)  # False
has_close_elements([1.0, 1.2, 3.0], 0.5)  # True
```

## 1.2. Model çıktısı

```python
def has_close_elements(numbers, threshold):
    for i in range(len(numbers)):
        for j in range(i + 1, len(numbers)):
            if abs(numbers[i] - numbers[j]) < threshold:
                return True
    return False
```

Bütün gizli testleri geçerse madde puanı **1**'dir.

## 1.3. Hatalı ama ikna edici çözüm

```python
if abs(numbers[i] - numbers[j]) <= threshold:
```

Benchmark `< threshold` beklerken model `<= threshold` kullanmışsa sınır değer testinde hata verir. Orijinal HumanEval'in az sayıda test içermesi bu tür sınır-durum hatalarını **kaçırma** riskini artırır — tam da Bölüm 3'te ele alınan EvalPlus'ın çözmeye çalıştığı sorun budur.

## 1.4. 2026'da doygunluk

Frontier modeller HumanEval'de **%97 ve üzeri pass@1** almaktadır. 97 ile 98 arasındaki fark, 164 sorudan yalnızca ~1,6 soruya denk gelir; bu da HumanEval'i **frontier modelleri birbirinden ayırt etmek için artık kullanışsız** kılar. 2026 itibarıyla HumanEval çoğunlukla küçük/orta ölçekli modelleri (7B-30B aralığı, hâlâ ayrım gücü olan) test etmek veya hızlı regresyon kontrolü için kullanılır.

# 2. MBPP

MBPP (Mostly Basic Python Problems), yaklaşık **974** başlangıç düzeyi Python görevi içerir. Her görev doğal dil açıklaması, örnekler ve testlerle tanımlanır.

## 2.1. Örnek

Görev:

```text
Bir listedeki tek sayıların karelerini döndüren fonksiyon yaz.
```

Beklenen:

```python
f([1, 2, 3]) == [1, 9]
```

Model:

```python
def f(xs):
    return [x * x for x in xs if x % 2 == 1]
```

Testleri geçerse **1** puan alır.

MBPP de HumanEval gibi 2026'da doygunlaşmıştır (frontier modeller **%97+**); bu nedenle çoğu güncel model kartı artık MBPP yerine LiveCodeBench, SWE-bench ailesi veya Aider Polyglot gibi daha zor/güncel testleri öne çıkarır.

# 3. EvalPlus, HumanEval+ ve MBPP+

Orijinal HumanEval ve MBPP testleri bazen yetersizdir: yanlış bir program, birkaç kolay/örnek testi tesadüfen geçebilir.

## 3.1. Kaynak

**EvalPlus** (Liu, Xia ve ark., "Is Your Code Generated by ChatGPT Really Correct? Rigorous Evaluation of Large Language Models for Code Generation", arXiv:2305.01210, NeurIPS 2023), LLM ve mutasyon tabanlı bir test üretici kullanarak HumanEval'in test sayısını **tam olarak 80 kat** artırır (`HumanEval+`) ve ayrıca veri kümesindeki hatalı referans çözümleri düzeltir.

## 3.2. Yayımlanmış sonuç

26 popüler LLM üzerinde yapılan kapsamlı değerlendirmede, yeni (genişletilmiş) test setindeki pass@k skorlarının, farklı k değerleri için **orijinal HumanEval'e kıyasla %19,3 ile %28,9 arasında düştüğü** gösterilmiştir. Bu, "doğru" sayılan kodun önemli bir kısmının aslında yalnızca **yetersiz test kapsamı** yüzünden geçtiğini kanıtlar.

## 3.3. Örnek

Model fonksiyonu normal listelerde doğru çalışıyor, fakat:

- Boş liste,
- Tek elemanlı liste,
- Negatif sayılar,
- Aynı sayıların tekrarları,
- Çok büyük değerler

gibi durumlarda bozuluyor olabilir.

Orijinal HumanEval:

```text
Geçti → 1
```

HumanEval+:

```text
Ek sınır testi başarısız → 0
```

Bu yüzden günümüzde `HumanEval+` sonucu, yalnızca `HumanEval` sonucundan **daha güvenilir** kabul edilir; birçok model kartı artık ikisini yan yana raporlar (ve aralarındaki fark, modelin ne kadar "sınır durumlara dayanıklı" kod ürettiğinin bir göstergesidir).

# 4. LiveCodeBench

**LiveCodeBench**, LeetCode, AtCoder ve Codeforces gibi platformlardan yeni yayımlanan yarışma sorularını **sürekli ekleyerek** veri sızıntısını azaltan, "kontaminasyonsuz" (contamination-free) bir benchmark olarak tasarlanmıştır.

Yalnızca kod üretimini değil:

- Kod çalıştırmayı (code execution),
- Program çıktısını tahmin etmeyi (output prediction),
- Hatalı kodu onarmayı (self-repair),
- Kendi çözümünü düzeltmeyi

de sınar — yani "tek atımlık" kod üretiminden çok daha geniş bir yetenek setini kapsar.

## 4.1. Örnek

Görev:

```text
Bir dizide toplamı en büyük ardışık alt dizinin toplamını bul.
```

Girdi:

```text
[-2, 1, -3, 4, -1, 2, 1, -5, 4]
```

Beklenen:

```text
6
```

Çünkü `[4, -1, 2, 1]` toplamı 6'dır.

Model kodu örnek testi geçse bile gizli testlerde boş dizi veya tümü negatif dizi üzerinde bozulursa yanlış sayılır.

## 4.2. Puan

200 sorunun 120'sinde ilk çözüm geçiyorsa:

\[
pass@1=120/200=\%60
\]

Beş denemeden en az biri 170 soruda geçiyorsa:

\[
pass@5=170/200=\%85
\]

Bunlar ayrı ayrı raporlanmalıdır.

## 4.3. Liderlik tablosu görüntüsü (Temmuz 2026)

Üçüncü taraf toplayıcı benchlm.ai'nin LiveCodeBench v6 anlık görüntüsüne göre (Temmuz 2026):

| Sıra | Model | pass@1 |
|---|---|---|
| 1 | Sakana Fugu-Ultra | %93,2 |
| 2 | Sakana Fugu | %92,9 |
| 3 | Kimi K2.6 | %89,6 |

**Not:** Bu, akademik olarak sabitlenmiş bir "yayımlanmış sonuç" değil, belirli bir tarihte alınmış bir liderlik tablosu görüntüsüdür (snapshot); LiveCodeBench sürekli güncellendiği için sıralama haftalar içinde değişebilir. DeepSeek-V3, Qwen3-Coder ve DeepSeek-R1 gibi açık ağırlıklı modeller akıl yürütme zinciri etkinleştirildiğinde farkı hızla kapatmaktadır, ancak Temmuz 2026 itibarıyla listenin en tepesinde henüz açık ağırlıklı bir model yoktur.

# 5. SWE-bench ve SWE-bench Verified

SWE-bench, küçük bir fonksiyon yazdırmak yerine **gerçek GitHub depolarındaki sorunları** çözdürür.

Orijinal sürüm:

- **12** Python deposundan,
- **2.294** gerçek GitHub issue ve pull request'inden

oluşur. Model tüm kod deposunu incelemeli, ilgili dosyaları bulmalı ve bir yama (patch) üretmelidir.

## 5.1. Görev biçimi

Bir issue şöyle olabilir:

```text
Belirli koşullarda Django QuerySet yanlış SQL üretiyor.
Mevcut davranışı düzeltin ve geriye dönük uyumluluğu bozmayın.
```

Model:

1. Depoyu arar.
2. İlgili sınıfı bulur.
3. Kod değişikliği yapar.
4. Testleri çalıştırır.
5. Gerekirse yamayı düzeltir.

## 5.2. Puanlama

İki test grubu önemlidir:

- **FAIL_TO_PASS:** Daha önce başarısız olan testler artık geçmeli.
- **PASS_TO_PASS:** Daha önce geçen testler bozulmamalı.

Model yeni testi düzeltmiş ama mevcut 3 testi bozmuşsa görev çözülmüş sayılmaz.

500 görevden 175'i tam çözülmüşse:

\[
Resolved=175/500=\%35
\]

## 5.3. SWE-bench Verified

**Verified** sürümü (Ağustos 2024'te tanıtıldı), profesyonel geliştiriciler tarafından çözülebilir ve doğru tanımlanmış olduğu kontrol edilen **500** görevden oluşur — orijinal SWE-bench'teki bazı bozuk/belirsiz/çözülemez görevleri elemek için oluşturulmuştur.

Uzun süre SWE-bench Verified, kodlama ajanlarının standart karşılaştırma noktası oldu; GitHub Copilot, Cursor, Claude Code ve OpenAI Codex gibi ürünler pazarlamalarında bu skoru kullandı.

# 6. 2026'nın büyük dönüşü: OpenAI, SWE-bench Verified'dan SWE-bench Pro'ya, sonra geri

Bu bölüm, benchmarkların ne kadar hızlı "eskiyebileceğine" dair 2026'da yaşanmış, oldukça öğretici ve tam anlamıyla güncel (bu belgenin yazıldığı hafta gerçekleşmiş) bir vakadır.

## 6.1. Birinci adım: SWE-bench Verified'dan uzaklaşma (2026 başı)

OpenAI, 2026 başında SWE-bench Verified'ı frontier kodlama yeteneğinin **temel göstergesi olarak artık kullanmadığını** açıkladı. Öne sürülen gerekçeler:

- **Doygunluk:** Skorlar liderlik tablosunun tepesinde kümelenmiş, modeller arası ayrım gücü kalmamıştı. OpenAI'nin ifadesiyle: *"Yüksek performans seviyelerinde, benchmark kazanımları artık gerçek dünya yetenek gelişimini güvenilir biçimde yansıtmıyor."*
- **Kirlenme riski:** *"SWE-bench Verified ve ilgili depolar hem açık kaynaklı hem de yaygın biçimde kullanılıyor ve tartışılıyor; bu da kirlenmeden kaçınmayı zorlaştırıyor."*
- **Metriğe yönelik optimizasyon şüphesi:** Küçük artışların gerçek yetenek gelişiminden çok, *"metriğe yönelik optimizasyon, eğitim sırasında benzer görevlere maruz kalma veya veri kümesi örtüşmesi"* yansıtabileceği belirtildi.

Ohio State Üniversitesi'nden Yu Su bu geçişi şöyle yorumladı: *"Her benchmarkın bir raf ömrü vardır ve bu, alanın ilerlemesinin bir yansımasıdır."*

OpenAI bu adımda, Scale AI araştırmacıları tarafından geliştirilen ve kamuya açık, elenmiş ve ticari kod tabanlarından oluşan **1.865 uzun-ufuklu (long-horizon) görev** içeren **SWE-bench Pro**'ya yönelmeyi önerdi — kirlenmeye dirençli bir test yatağı olarak tanıtıldı (Scale AI, arXiv:2509.16941).

## 6.2. İkinci adım: SWE-bench Pro'nun kendisi de sorunlu çıktı (8 Temmuz 2026)

Yalnızca birkaç ay sonra, **8 Temmuz 2026**'da OpenAI resmî X (Twitter) hesabından şu açıklamayı yayımladı:[^swebenchpro]

> "We audited SWE-Bench Pro, one of the most widely used AI coding benchmarks, and found it no longer reliably measures frontier coding capability. We find 30% of SWE-Bench Pro tasks to be broken, and are retracting our previous recommendation that the research community use it..."

## 6.3. Yayımlanmış sonuç: denetim bulguları

OpenAI'nin kendi "datapoint-analysis" hattı, SWE-bench Pro'nun **731 görevlik açık (public) bölümünü** inceledi ve sonuçları **beş deneyimli yazılım mühendisinin bağımsız incelemesiyle** çapraz doğruladı. Bulgular:

- Görevlerin **yaklaşık %30'u bozuk** çıktı.
- En yaygın hata türü: **aşırı katı testler** — görevde hiç belirtilmemiş uygulama detaylarını zorunlu kılan testler, işlevsel olarak doğru çözümleri bile başarısız sayıyordu.
- Bazı görevler, gerçekten doğru kod yazmak yerine **bozuk bir testi "kandırarak"** geçilebiliyordu.

Bu, Şubat 2026'da yapılmış tavsiyenin **beş aydan kısa sürede** geri çekilmesi anlamına geliyordu — sektörün GitHub Copilot, Cursor, Claude Code ve OpenAI Codex gibi ürünleri pazarlarken SWE-bench skorlarına ne kadar dayandığı düşünüldüğünde, zamanlaması dikkat çekiciydi.

## 6.4. Bu vakadan çıkarılacak ders

```text
2024        SWE-bench Verified → standart hâline gelir
2026 Şubat  OpenAI: "Verified artık ayrım gücü sunmuyor, kirlenme riski yüksek"
            → SWE-bench Pro öneriliyor
2026 Temmuz OpenAI: "SWE-bench Pro'nun %30'u bozuk"
            → Önceki tavsiye geri çekiliyor
```

**Sonuç:** Tek bir kodlama benchmark skoru, hatta iyi tanıtılmış ve büyük laboratuvarlarca önerilmiş biri bile, sürekli sorgulanmalıdır. "Model X, SWE-bench Y'de %72 aldı" cümlesi; benchmark'ın hangi sürümü, ne zaman denetlendiği ve hâlâ güvenilir kabul edilip edilmediği belirtilmeden eksik kalır.

# 7. SWE-bench Multimodal ve Multi-SWE-bench

## 7.1. SWE-bench Multimodal

SWE-bench Multimodal, sorun tanımlarına **görsel bağlam** (ekran görüntüleri, tasarım mockup'ları) ekleyen bir varyanttır; modelin kod üretirken görsel bilgiyi kullanıp kullanamadığını test eder. 13 Ocak 2025'te ana SWE-bench deposuna entegre edilmiştir. Skorlar, yalnızca metin tabanlı SWE-bench varyantlarına kıyasla **belirgin biçimde düşüktür** — bu da görsel bağlamı kod değişikliğine dönüştürmenin, saf metin tabanlı issue çözmekten daha zor bir yetenek olduğunu gösterir.

## 7.2. Multi-SWE-bench

**Multi-SWE-bench** (arXiv:2504.02605), SWE-bench'in Python-merkezli olmasının bir eleştirisi olarak doğmuştur: **7 farklı programlama dilinde**, insan tarafından doğrulanmış **1.632 GitHub örneği** içerir. Amaç, "bir model Python'da iyi issue çözüyor ama Go veya Rust'ta çok daha kötü" gibi dile-özgü yetenek farklarını ortaya çıkarmaktır.

### Model çıktısı (temsilî)

| Dil | Resolved oranı (temsilî) |
|---|---|
| Python | %38 |
| Go | %24 |
| Rust | %19 |
| Java | %27 |

Bu tablo, tek dilli SWE-bench sonuçlarının çok dilli gerçek dünya yazılım geliştirmeye genellenemeyebileceğini gösteren **temsilî** bir örnektir.

# 8. DS-1000 ve DS-Bench

## 8.1. DS-1000

**DS-1000** (arXiv:2211.11501), gerçek StackOverflow sorularından derlenmiş **1.000** veri bilimi kod problemi içerir. Kapsadığı yedi kütüphane:

- NumPy, pandas, SciPy, scikit-learn, Matplotlib, PyTorch, TensorFlow.

Kod hem işlevsel testlerden hem de bazı yüzeysel kısıtlardan (ör. belirli bir API'nin kullanılıp kullanılmadığı) geçirilir. Yayımlandığı dönemde en iyi sistem olan **Codex-002, %43,3** doğruluk almıştır — bu, veri bilimi kodunun genel amaçlı kod üretiminden neden daha zor olduğunu gösteren erken bir kanıttı.

### Örnek

Görev:

```text
pandas DataFrame'de her müşteri için en son tarihli satırı seç.
```

Beklenen yaklaşım:

```python
df.sort_values("date").groupby("customer").tail(1)
```

Model yalnızca genel olarak "groupby kullanın" diye açıklama yaparsa, çalıştırılabilir kod üretmediği için **0** alır.

## 8.2. DS-Bench

**DS-Bench** (arXiv:2505.15621, 2025), DS-1000'in basitliğine yönelik bir eleştiri olarak GitHub kaynaklı, daha gerçekçi **1.000** problem ve **10** kütüphane ile kapsamı genişletmiştir.

Temel fark, görev karmaşıklığındadır: DS-1000'in ortalama çözüm uzunluğu **3,6 satır** iken, DS-Bench'in ortalama çözüm uzunluğu **22,5 satırdır** — yani DS-Bench, StackOverflow'daki tek satırlık "nasıl yapılır" sorularından çok, gerçek bir veri bilimi iş akışına daha yakın, çok adımlı görevler içerir.

# 9. Aider Polyglot

**Aider Polyglot** (Aralık 2024'te tanıtıldı, `aider.chat/docs/leaderboards/`), kod **düzenleme** (editing) becerisini — yalnızca sıfırdan kod üretmeyi değil, mevcut bir kod tabanında **doğru formatta değişiklik uygulamayı** — ölçen bir benchmarktır. SWE-bench'in Python ağırlıklı olmasına karşı bir tepki olarak, bilinçli biçimde çok dilli tasarlanmıştır.

## 9.1. Yapı

- **697** Exercism programlama alıştırmasından, en az **5/7 temel modelin başarısız olduğu** en zor **225** alıştırma seçilmiştir.
- **6 dil:** C++, Go, Java, JavaScript, Python, Rust.
- **İki deneme protokolü:** Model ilk denemede başarısız olursa, birim testlerinin hata çıktısı kendisine gösterilir ve ikinci bir deneme hakkı verilir. Bu, modelin yalnızca "ilk seferde doğru yazma" değil, **hata geri bildirimini kullanarak kendini düzeltme** becerisini de ölçer.
- Birincil metrik **`pass_rate_2`**: ikinci denemeden sonra 225 alıştırmanın kaçında tüm gizli testlerin geçtiğinin yüzdesi.

## 9.2. Liderlik tablosu görüntüsü (Temmuz 2026)

- **GPT-5** liderlik tablosunun başında, **%88,0** (0,880) skorla, 22 değerlendirilmiş model arasında.
- Açık ağırzlıklı modeller arasında en iyisi **DeepSeek-V3.2-Exp**, **%74,5** (0,745) ile genel sıralamada 5. sırada.
- Claude Opus 4.6, Mayıs 2026 itibarıyla benchmark genelinde tutarlı biçimde güçlü performans göstermiştir.

**Not:** Bu sayılar aider.chat/llm-stats.com'un Temmuz 2026 anlık görüntüsüdür; liderlik tablosu her yeni model sürümünde güncellenir.

# 10. METR: "zaman ufku" (time horizon) metodolojisi

Bugüne kadar anlatılan benchmarkların hepsi "görev tamamlandı mı, tamamlanmadı mı" sorusuna odaklanır. **METR** (Model Evaluation and Threat Research), farklı bir soru soruyor: *"Bir AI ajanı, ne kadar UZUN süren görevleri güvenilir biçimde tamamlayabilir?"*

## 10.1. Metodoloji

METR'in temel metriği **görev-tamamlama zaman ufku** (task-completion time horizon): bir AI ajanının, belirli bir güvenilirlik düzeyinde başarılı olduğu görev **uzunluğu**, aynı görevi bir insan uzmanın tamamlaması için gereken süreye göre ölçülür.

- **%50 zaman ufku:** Ajanın görevlerin yarısında başarılı olduğu görev süresi.
- Metodoloji, **170 görevlik** bir test paketine dayanır: yazılım mühendisliği, siber güvenlik, genel akıl yürütme ve makine öğrenmesi görevlerini kapsar; bu görevler, AI ajanlarının gerçek AI araştırmasına ne kadar katkı sağlayabileceğini yansıtacak şekilde derlenmiştir.

## 10.2. Yayımlanmış sonuç

METR'in bulgularına göre:

- Bu metrik, son **6 yılda tutarlı biçimde katlanarak artmıştır**; **50% zaman ufku yaklaşık her 7 ayda bir iki katına çıkmıştır** — **2019'da 4 saniyeden**, **2026'da 16 saatin üzerine** çıkmıştır.
- **2024-2025 döneminde bu ivme hızlanmıştır**: ikiye katlanma süresi 7 aydan yaklaşık **4 aya** düşmüştür.

## 10.3. Model çıktısı (kavramsal örnek)

```text
2019: GPT-2 sınıfı bir model → %50 güvenilirlikle ~4 saniyelik görevler
2023: GPT-4 sınıfı bir model → %50 güvenilirlikle ~dakikalar süren görevler
2025: Frontier ajan modeli   → %50 güvenilirlikle ~saatler süren görevler
2026: Frontier ajan modeli   → %50 güvenilirlikle 16+ saatlik görevler
```

Bu, "pass@1 %80" gibi statik bir sayı yerine, **yeteneğin zaman içindeki artış eğrisini** doğrudan gösteren, agentic (ajan tabanlı) kodlama yeteneğini ölçmek için giderek daha çok referans gösterilen bir yaklaşımdır.

# 11. Terminal-Bench

**Terminal-Bench** (arXiv:2601.11868), AI ajanlarını gerçek bir **terminal ortamında** test eder: kod derleme, küçük bir model eğitme, sunucu yapılandırma, hata ayıklama veya sistem yönetimi gibi gerçek iş akışlarından esinlenen görevler.

## 11.1. Yapı

- **Terminal-Bench 2.0**, özenle seçilmiş **89 görevden** oluşur; her görev özgün bir ortam, insan tarafından yazılmış bir referans çözüm ve doğrulama için kapsamlı testler içerir.
- Her görev izole bir **Docker sandbox** içinde çalışır: ajan gerçek bir shell'i yönetir, çıktıyı gözlemler ve yinelemeli olarak ilerler.
- Başarı, ajanın **transkriptine değil**, ortamın **son durumunu** kontrol eden otomatik doğrulama testleriyle belirlenir — yani "doğru şeyi söyledi mi" değil "sistemi gerçekten doğru duruma getirdi mi" ölçülür.
- Kapsanan alanlar: bilimsel hesaplama, yazılım mühendisliği, makine öğrenmesi, güvenlik, sistem yönetimi, veri bilimi.

## 11.2. Yayımlanmış sonuç

Frontier modeller ve ajanlar bu benchmark'ta **%65'in altında** skor almaktadır — bu, Terminal-Bench'i 2026 itibarıyla hâlâ **doygunlaşmamış**, ayrım gücü yüksek bir agentic-kodlama testi yapar.

### Model çıktısı (temsilî görev)

```text
Görev: "Bir PostgreSQL veritabanı kurup, örnek veriyle doldurup,
belirli bir sorguyu 200ms altında çalışacak şekilde indeksleyin."

Ajan adımları:
$ apt install postgresql
$ psql -c "CREATE DATABASE demo;"
$ psql demo < seed_data.sql
$ psql demo -c "EXPLAIN ANALYZE SELECT ..."
$ psql demo -c "CREATE INDEX ..."
$ psql demo -c "EXPLAIN ANALYZE SELECT ..."  # doğrulama
```

Doğrulama testi, sorgunun gerçekten hedeflenen sürede çalışıp çalışmadığını otomatik kontrol eder; ajan "indeks oluşturdum" dese bile sorgu hâlâ yavaşsa görev **başarısız** sayılır.

# 12. Rekabetçi programlama: Codeforces ELO tarzı değerlendirme

Kod benchmarklarının bir diğer türü, modelleri doğrudan **insan yarışmacılarla aynı ölçekte** (Codeforces ELO reytingi gibi) değerlendirmektir.

## 12.1. Yayımlanmış sonuç: OpenAI o-serisi ilerlemesi

OpenAI'nin "Competitive Programming with Large Reasoning Models" (arXiv:2502.06807) makalesinde raporlanan Codeforces reyting ilerlemesi:

| Model | Codeforces reytingi | Yaklaşık yüzdelik dilim |
|---|---|---|
| o1-preview | 1258 | ~62. persentil |
| o1 | 1673-1891 | ~89. persentil |
| o1-ioi (IOI'ye özel ayarlı) | 2214 | ~98. persentil |
| o3 | 2724-2727 | ~%99,2-99,8 persentil |

Referans noktası: **2400 Elo eşiği**, insan rekabetçi programcılarının yaklaşık **%99,2'lik** dilimine denk gelir. Ayrıca o3, **2024 IOI'de (Uluslararası Bilgi Olimpiyatı)**, göreve özel el yapımı stratejiler veya gevşetilmiş kısıtlar kullanılmadan **altın madalya** düzeyine ulaşmıştır — genel amaçlı, ölçeklendirilmiş bir modelin, alana özel ayarlanmış özel sistemleri geride bıraktığı bir örnek olarak vurgulanmıştır.

## 12.2. ICPC 2025: iki laboratuvarın altın madalyası

Matematik olimpiyatlarındaki (bkz. `02_matematik_benchmarklari.md`, Bölüm 11) IMO 2025 vakasına paralel biçimde, **ICPC 2025 Dünya Finali**'nde hem Google DeepMind (Gemini) hem de OpenAI modelleri **altın madalya düzeyinde** performans sergilediğini duyurdu; OpenAI kendi modelinin **kusursuz** (mükemmel) sonuç aldığını iddia etti. Bu, matematik olimpiyatlarında olduğu gibi, "resmî değerlendirme mi yoksa şirket-içi değerlendirme mi" ayrımının rekabetçi programlama sonuçları için de aynı dikkatle okunması gerektiğini gösterir.

## 12.3. Neden önemli?

Codeforces/ICPC/IOI tarzı değerlendirme, statik bir "pass@1" sayısından farklı olarak modelin performansını **doğrudan insan yeteneği dağılımı içine** yerleştirir — "model X, insan yarışmacıların %99,2'sinden daha iyi" gibi bir ifade, soyut bir yüzdeden çok daha sezgisel bir referans çerçevesi sunar. Fakat aynı zamanda, bu tür yarışma sorularının **halka açık** olması (Codeforces arşivleri herkese açıktır) kirlenme riskini de beraberinde getirir — bu nedenle güncel değerlendirmeler genellikle **henüz yapılmamış/çok yeni** yarışmaları tercih eder (LiveCodeBench'in felsefesiyle aynı mantık).

# 14. Genel çıkarımlar

1. **Fonksiyon-düzeyi testler (HumanEval, MBPP) 2026'da doygun.** %97+ skorlar artık frontier modelleri ayırt etmiyor; bu testler küçük modeller veya hızlı regresyon kontrolü için hâlâ anlamlı.

2. **Test kapsamı, testin kendisi kadar önemli.** EvalPlus'ın gösterdiği gibi, yetersiz test seti "doğru" görünen ama aslında hatalı kodu ödüllendirebilir — pass@k'nin **hangi** test setiyle hesaplandığı belirtilmeden sayı eksik kalır.

3. **Repo-düzeyi (SWE-bench ailesi) testler, benchmarkların ne kadar hızlı eskiyebileceğinin en çarpıcı örneği.** 2026'nın ilk yedi ayında yaşanan "Verified → Pro → Pro'dan geri çekilme" döngüsü, tek bir sabit sayıya güvenmenin riskini gösteriyor.

4. **Agentic/ajan yeteneği, "pass@1" ile tam yakalanamıyor.** METR'in zaman ufku yaklaşımı ve Terminal-Bench'in uçtan uca doğrulaması, modern kodlama ajanlarını değerlendirmek için "tek seferlik doğru cevap" mantığından **"ne kadar uzun ve karmaşık bir işi güvenilir tamamlayabiliyor"** mantığına geçişi temsil ediyor.

5. **Rekabetçi programlama/ELO tarzı değerlendirme sezgisel ama dikkatli okunmalı.** "İnsanların %99,2'sinden iyi" gibi ifadeler güçlü görünür, fakat resmî/bağımsız doğrulama olup olmadığı (ICPC/IOI vakalarında olduğu gibi) mutlaka kontrol edilmelidir.

6. **Çok dillilik gözden kaçırılan bir boyut.** SWE-bench ve HumanEval'in Python ağırlıklı olması, Aider Polyglot ve Multi-SWE-bench'in gösterdiği gibi, modellerin diğer dillerdeki gerçek performansını gizleyebilir.
